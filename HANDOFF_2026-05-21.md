### PR #54 — Remove unused diagnostic endpoints (`7981d97`)
- Deleted three internal diagnostic API routes that were no longer used and had **no automated callers** (the integrity cron runs the Python `sentinel-run-checks` against the DB directly, not over HTTP). Also removed a stale endpoint reference from the alert-email body.

### PR #55 — Default-deny authentication for the API surface (`86364c7`)
- Inverted `apps/web/proxy.ts` (Next.js 16 renamed middleware to **Proxy**) from allowlist-protect to **deny-by-default**: every `/api` route requires a signed-in Clerk session **except** an explicit public allowlist (currently just the Stripe webhook, which authenticates by signature). Unauthenticated API requests get a JSON **401**.
- Page gating is unchanged (`/app` behind sign-in, `/app/feed` public). New API routes are now protected by construction rather than relying on a protect list.

## Database (live Supabase project: "Sentinel Map")
Applied to the live database and verified, and captured as repo migrations under `packages/db/migrations/`:
- **`0009_watches.sql`** — per-user watchlist table (keyed by `clerk_user_id`, FK to `raw_posts`).
- **`0010_enable_rls.sql`** — Row-Level Security enabled on all public tables. The app connects with a role that bypasses RLS, so server-side queries are unaffected; the Supabase Data API roles (`anon`/`authenticated`) are denied by default.
- **`0011_harden_db.sql`** — pinned `search_path` on `confidence_score` and `events_in_bbox`; removed Data API access from the `source_reliability` materialized view. (The role-specific `REVOKE` is guarded so the migration is safe on non-Supabase Postgres.)

Supabase security advisors are clean except the expected, benign "RLS enabled, no policy" INFO notices (that is the intended deny-by-default state).

### Recommended future hardening (prioritized)
1. **Least-privilege DB roles.** The app currently connects with a broad database role. Introduce scoped roles — `app_web` (read-mostly; writes only `watches` and `user_subscriptions`) and `app_ingest` (pipeline writes) — as `NOSUPERUSER` with no DDL and `BYPASSRLS`, then point each consumer's `DATABASE_URL` at the right role and keep the privileged role for migrations only. Validate with a `SET ROLE` dry-run on a Supabase branch before cutover. (A full step-by-step plan was produced in the 2026-05-21 session.)
2. **Decide on the Supabase Data API.** The app talks to Postgres directly and does not use the Data API, so consider disabling it entirely (removes the anon-key surface) — or, if client-side Supabase is ever adopted, write per-user RLS policies first (especially on `user_subscriptions` and `watches`).
3. **CI quality gate + branch protection on `main`.** Add a required CI job (`tsc --noEmit`, `eslint`, tests) and enable secret-scanning / push protection; require PR + green checks before merge. Note: the web app currently has **no test harness** — standing one up is the natural place to lock the auth policy from #55 with a `proxy` test.
4. **One migration source of truth.** Schema changes currently flow through both the repo's `migrate.py` and Supabase's own migration history; consolidate to avoid drift.
