## Root cause — GDELT 2.0 column-index bug
`GdeltEventsIngestor` downloads the latest 15-min GDELT 2.0 **export** file and filters rows
**client-side by column index**. The constants were wrong for the 61-column Event layout:

| constant | was | actually read | correct |
|---|--:|---|--:|
| `_COL_COUNTRY` | 51 | `ActionGeo_Type` (int 0–5) | **53** (`ActionGeo_CountryCode`) |
| `_COL_DATE_ADDED` | 1 | `SQLDATE` (8-digit) | **59** (`DATEADDED`) |
| `_COL_GEO_NAME` | 53 | `ActionGeo_CountryCode` | **52** |
| `_COL_LAT` | 55 | `ActionGeo_ADM2Code` | **56** |
| `_COL_LNG` | 56 | `ActionGeo_Lat` | **57** |
| `_COL_SOURCE_URL` | 57 | `ActionGeo_Long` | **60** |

`row[51]` (an int 0–5) compared against `"UP"/"IR"/…` never matched → 100% of rows skipped for every
theater; independently, `DATEADDED` read from `SQLDATE` failed the `yyyymmddHHMMSS` parse. Either alone
yields `count:0`. `gdelt_ukraine` at 0 was the tell — impossible if parsing were correct.
