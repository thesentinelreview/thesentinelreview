## New Findings

### P1

**NEW-01 — `generate_briefing.py`: theater validation too late (P1)**
`apps/ingest/sentinel/jobs/generate_briefing.py`

`_compute_baseline(theater, conn)` uses `_THEATER_BBOX.get(theater)` and returns `{}` on unknown theater. The function is called AFTER event selection (`events = get_recent_events(...)`), so an invalid `params.theater` produces events + empty baseline, and briefing generation proceeds with stale baseline data rather than failing loudly. The `if not events:` guard checks event count, not baseline validity.
- **Fix sketch:** Validate `theater in _THEATER_BBOX` before event selection; if invalid, log error and return without generating.
- **Severity:** P1 — would silently produce a malformed briefing and mark it published.

### P2

**NEW-02 — D-4 root cause: bulk `recompute_source_health()` has no Python-layer fallback (P2)**
`apps/ingest/sentinel/db.py:229`, `packages/db/migrations/0028_source_health_function.sql:94-106`

Per-fetch `record_source_fetch()` calls `recompute_source_health(source_id)` — single-source mode that trusts the forward stamp and skips raw_posts reconciliation (migration 0028 line 51). The bulk mode (`p_source_id IS NULL`, which reconciles `last_post_at` from `raw_posts`) is scheduled only via `pg_cron` in migration 0028 (line 96-101); it is **not called from `runner.run_ingest()` or any Python scheduler entry point**. If pg_cron's scheduled job fails or isn't installed (dev/preview), 20 sources with NULL `last_post_at` remain permanently in `health_status='unknown'` even when they have raw_posts — because the per-fetch path cannot reconcile from `raw_posts`. The `scheduler.py` has no awareness of the bulk recompute.
- **Fix sketch:** Add `SELECT recompute_source_health()` (bulk, NULL arg) to `runner.run_ingest()` after the drain, or add it as a Python-scheduler periodic job. Provides belt-and-suspenders alongside pg_cron.
- **Note:** This is the D-4 bug. See the data-audit question answers below for full analysis.

**NEW-03 — `account/page.tsx`: Stripe API failure silently serves stale renewal date (P2)**
`apps/web/app/account/page.tsx` (account page)

`liveRenewalDate()` catches all exceptions from the Stripe API and returns the DB fallback `Date` without any user-visible signal. A Stripe auth failure (expired key, rate limit) silently shows an outdated renewal date to the subscriber.
- **Fix sketch:** Log the error with structlog; consider surfacing a UI indicator ("last updated [date]") so the user can tell the date is stale.

**NEW-04 — `extract_events.py`: dedup audit trail silently fails on insert error (P2)**
`apps/ingest/sentinel/jobs/extract_events.py` (`_record_dedup_decision_safe`)

The dedup-decision insert is wrapped in a broad `except Exception` that logs a warning but continues. A persistent schema or permission error would silently corrupt the audit trail (dedup_decisions underpopulated) with no operational alert. The function name says "safe" but the failure mode isn't surfaced to the integrity watchdog.
- **Fix sketch:** Distinguish transient (retry) vs. permanent (escalate via log.error with structured event_id context) errors; add a counter in the run summary so operators see `dedup_decision_failures=N`.

**NEW-05 — `theaterLabelCase()` missing ELSE clause (P2)**
`apps/web/lib/queries.ts:191`

```typescript
return `CASE ${whens.join(" ")} END`;  // no ELSE
```

Returns SQL `CASE … END` with no `ELSE`. An event outside all 5 theater bboxes gets `NULL` as its `bbox_theater` label. In practice `theaterPredicate` pre-filters events so NULL shouldn't occur, but the inconsistency exists: the API's `THEATER_CASE_SQL` (`api-v1.ts:104`) has `ELSE 'other'` while the tie-out path silently NULLs. Becomes a P1 when the Russia theater is added and the bbox definitions temporarily diverge during deployment.
- **Fix sketch:** Add `ELSE 'unknown'` (or `'other'` to match the API) to `theaterLabelCase()`.

## Data-Audit Question Answers

### D-1 — Theater coverage gap mechanism (confirm only; fix is separate ticket)

**Confirmed: bbox boundary gap, not a routing or bucketing bug.**

`theater_router.py:59` system prompt explicitly routes "strikes inside Russia" to `ukraine` theater: *"ukraine: the Russia–Ukraine war (including strikes inside Russia and Black Sea naval action)."* The LLM correctly assigns Moscow/St. Pete/oil-refinery events to `ukraine`. The `ukraine` bbox `[22, 44, 40, 52]` (lng_min, lat_min, lng_max, lat_max) covers up to 52°N. Moscow is at 55.7°N — north of the box. St. Petersburg (60°N), Yaroslavl (57°N), and Ust-Luga (59°N) are similarly north.

No systematic mis-bucketing in the remaining 5 theaters found:
- Israel carve-out from Iran bbox works correctly (`israelCarveOut()` in queries.ts:53 and `_iran_israel_carve_sql()` in db.py).
- `theaterLabelCase()` precedence order (israel > ukraine > iran > sudan > myanmar) correctly handles the israel/iran overlap.
- No events bucketed into the wrong theater via the routing logic — the issue is purely the bbox not covering far-north Russia.

**Other gaps:** None found beyond the ~170 deep-Russia events. Sudan, Myanmar, Iran coverage is adequate. The 2-degree lat gap between ukraine (max 52°N) and iran (max 42°N) means events in ~42–44°N Caucasus / Central Asia are unbucketed, but no OSINT sources currently cover those events.

### D-2 — `confidence` vs. `has_strong_signal` inconsistency

**Finding: INTENDED BEHAVIOR. Not a bug. The two fields measure different things.**

**Code path:**

`scorer.py:classify()` (lines 19-40) requires:
- `source_count >= 2 AND platform_count >= 2 AND strong_signal` → `verified`
- `source_count >= 2 AND platform_count >= 2` → `partial`
- `source_count >= 2 AND strong_signal` → `partial`
- `source_count == 1 AND tier == 1 AND strong_signal` → `partial`
- Otherwise → `unconfirmed`

`scorer.py:has_strong_signal()` (lines 43-59) returns `True` if ANY single geolocation indicator is present: `geolocated_footage`, `coordinates_given`, `landmarks_visible`, `official_acknowledgment`, or `matching_press`.

**The 402 `unconfirmed + has_strong_signal=true` events:** Single-source, non-tier-1 posts with strong geolocation evidence (e.g., a Bluesky tier-2 account posting geolocated footage of a strike). `classify()` returns `"unconfirmed"` for these — correct per the model design. The strong signal exists, but cross-platform corroboration does not.

**The 233 `partial + has_strong_signal=false` events:** Multi-source cross-platform events without any of the five geolocation markers. `classify()` returns `"partial"` (`source_count >= 2 AND platform_count >= 2` — no strong_signal required). Also correct.

**BUG-007 relationship:** BUG-007 (recompute not running on bulk events) is an independent coverage gap — it can cause a *previously unconfirmed* event to remain unconfirmed after gaining a second source, but it doesn't explain why `unconfirmed + signal=true` or `partial + signal=false` can simultaneously occur. Both cross-tab patterns are intended outcomes of the current model.

**Recommendation:** No code fix for D-2. Thread copy should frame confidence as a *method* (confidence bands, corroboration scoring) and avoid implying a high verified rate (only 3% verified). Do not conflate with BUG-007, which is a separate open ticket.

### D-4 — NULL `last_post_at` forces `health_status='unknown'`

**Finding: CONFIRMED BUG. Root cause: per-fetch path doesn't reconcile from `raw_posts`; bulk path is pg_cron-only with no Python fallback.**

**Code path:**

`db.record_source_fetch()` (db.py:229) calls `recompute_source_health(source_id)` — **single-source mode**. Migration 0028, line 51: `IF p_source_id IS NULL THEN … UPDATE sources SET last_post_at = (max raw_posts.posted_at)`. The `IF` block is skipped when a specific `source_id` is passed. So per-fetch never reconciles from `raw_posts`.

Migration 0028, lines 94-106: schedules `SELECT recompute_source_health()` (bulk, NULL) via `pg_cron` at `'0 * * * *'` (hourly), guarded by `IF EXISTS pg_cron`. On production Supabase, pg_cron is enabled. The Python `scheduler.py` has no entry for bulk health recompute.

**Why 20 sources are still 'unknown':** Most likely one of:
1. Those 20 sources have no `raw_posts` at all (never had a successful ingest) → `'unknown'` is technically correct, but `is_active=true` is misleading.
2. Those 20 sources have `raw_posts` with `posted_at IS NULL` (non-RSS ingestors that don't populate `posted_at`) → bulk reconcile's `max(posted_at)` returns NULL → no update → still 'unknown'.
3. pg_cron hourly job has silent failures (out of scope to diagnose here).

**Fix sketch (D-4 ticket):** (a) Add `SELECT recompute_source_health()` call to `runner.run_ingest()` after the queue drains, as belt-and-suspenders alongside pg_cron; (b) ensure non-RSS ingestors (Telegram, X, Bluesky) correctly write `newest_posted_at` in their `_fetch_meta()` for the per-fetch forward stamp path; (c) run the bulk once immediately to reclassify the 20 sources and report breakdown.
