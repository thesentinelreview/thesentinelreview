## 3. Findings

### BUG-001 — `israel` theater is missing from every layer
Severity: P1   Domain: B   Confidence: confirmed
Location: `apps/ingest/sentinel/pipeline/theater_router.py:29` (and system prompt L51-57); `apps/web/lib/types.ts:4`; `apps/web/data/theaters.ts`; seed in `packages/db/migrations/0012_add_sources_2026_05.sql`
Evidence:
```python
# theater_router.py:29
THEATERS: tuple[str, ...] = ("ukraine", "iran", "sudan", "myanmar")
```
```typescript
// apps/web/lib/types.ts:4
export type TheaterKey = "ukraine" | "iran" | "sudan" | "myanmar";
```
Read-only query result: `sources` and `briefings` both contain only `{iran, myanmar, sudan, ukraine}`.
Impact: The product spec declares 5 theaters (Ukraine, Myanmar, Iran, Sudan, **Israel**). No layer treats Israel as a theater; the router's system prompt explicitly folds Israel into the `iran` theater ("Israel–Iran strikes…"). Consequently any Israel/Gaza-centric content is labeled `iran`, a user filtering for an Israel theater finds nothing, and the theater router cannot emit `israel` even if intended. This is consistent drift across all layers, so it is not corrupting existing rows — but it means an entire declared theater is uncovered.
Verification: `SELECT DISTINCT unnest(theaters) FROM sources;` (returns 4, no israel). Grep `israel` across `apps/` and `packages/` returns only prose mentions inside the `iran` theater copy.
Suggested direction: Confirm with the maintainer whether Israel is a real 5th theater or intentionally subsumed under `iran`; if real, add it consistently to `THEATERS`, `TheaterKey`, `data/theaters.ts`, badge/color maps, and seed.

### BUG-002 — Hard-coded `"ukraine"` theater fallback (default-theater leak)
Severity: P1   Domain: B   Confidence: probable
Location: `apps/ingest/sentinel/pipeline/theater_router.py:75-76, 100-103`
Evidence:
```python
prior = (source.get("theaters") or ["ukraine"])[0]        # L75
fallback = prior if prior in THEATERS else "ukraine"       # L76
...
else:
    # Missing/unexpected value (incl. no tool call): fail OPEN to the prior
    theater = fallback                                     # L102
    log.warning("theater_router_unexpected_choice", choice=choice, fallback=fallback)
```
Impact: This is the known "default theater" leak archetype. If a source has no `theaters` configured (empty/null array) **or** the value is invalid, `prior`/`fallback` silently becomes `"ukraine"`. On any router glitch (no tool call, unexpected enum value) the post fails open to that fallback. A Sudan/Myanmar/Iran source thereby has its posts routed to the Ukraine extraction prompt and labeled `ukraine`, mixing theaters. Combined with BUG-001, there is no path to ever label `israel`.
Verification: Inspect sources with empty `theaters`: `SELECT id, handle, theaters FROM sources WHERE theaters IS NULL OR array_length(theaters,1) IS NULL;` then trace one post through `classify_theater`.
Suggested direction: Make the fallback explicit-unknown rather than a real theater, or require non-empty `theaters` at the source layer; log+route glitches to a quarantine theater instead of `ukraine`.

### BUG-003 — `migrate.py` bootstrap skips pending migrations without running them
Severity: P2   Domain: C   Confidence: confirmed
Location: `packages/db/migrate.py:56-66`
Evidence:
```python
# When schema_migrations is empty but the DB is already initialised:
all_files = sorted(migration_files, key=lambda p: p.name)
to_bootstrap = all_files[:-1]          # marks every file except the LAST as applied
for path in to_bootstrap:
    conn.execute(
        "INSERT INTO schema_migrations (filename) VALUES (%s) ON CONFLICT DO NOTHING",
        (path.name,),
    )
```
Impact: The bootstrap branch fires whenever `schema_migrations` is empty but a `sources` table exists. It records **all but the final** migration file as applied *without executing them*, then only attempts the last. If a fresh/restored environment (or one where the tracking table was dropped) is bootstrapped while several genuinely new migrations are pending, every pending migration except the newest is permanently marked applied and never run — silent schema drift with no error. Not currently triggered (the live `schema_migrations` is populated through 0027), but it is a latent footgun on any DR/restore.
Verification: Static; reproduce by pointing `migrate.py` at a DB that has `sources` but an empty `schema_migrations` and 2+ pending files.
Suggested direction: On bootstrap, only mark migrations as applied after confirming their objects exist (or run all pending in order); never blanket-skip by filename.

### BUG-006 — Partial/rate-limited non-RSS fetches classify as healthy/silent
Severity: P2   Domain: E   Confidence: probable
Location: `apps/ingest/sentinel/ingestors/x.py:74-98`; `bluesky.py:115-127`; classifier `apps/ingest/sentinel/db.py:130-163`
Evidence:
```python
# x.py — 429 breaks the loop without setting transport_error
if resp.status_code == 429:
    log.warning("x_rate_limited", handle=handle)
    break
```
```python
# bluesky.py — mid-pagination error only records transport_error if NO results yet
except Exception as exc:
    log.error("bluesky_fetch_error", handle=handle, page=page, error=str(exc))
    if not results:
        transport_error = f"{type(exc).__name__}: {exc}"
    break
```
Impact: `_classify_fetch` maps a fetch to `erroring`/`url_broken` only when `transport_error` is set; otherwise it falls back to yield-based `healthy`/`silent`. On an X rate-limit, or a Bluesky failure after the first page, `transport_error` stays `None`, so a partially-failed or truncated fetch is stamped healthy/silent. This is the mechanism behind "feeds erroring while still classified healthy" for the social platforms.
Verification: Force a 429 (or a mid-pagination exception) in a test and assert the resulting `sources.health_status`.
Suggested direction: Set `transport_error` (or a dedicated `partial`/`rate_limited` flag) whenever pagination terminates abnormally, regardless of whether some results were already collected.

### BUG-007 — `has_strong_signal` / `confidence` have no recompute path outside corroboration insert
Severity: P2   Domain: A   Confidence: probable
Location: `apps/ingest/sentinel/jobs/extract_events.py:326-388` (and `:346`); column from `packages/db/migrations/0023_add_has_strong_signal_to_events.sql`
Evidence: `has_strong_signal` is set at insert and OR-in'd when a corroborating source attaches; `confidence` is re-`classify()`d on the same corroboration path. No trigger, pg_cron job, or scheduled worker recomputes either field. The only batch path is the optional one-shot `confidence_backfill.py` (manual GitHub Action).
Impact: Derived-state-set-once archetype. Any bulk SQL change to an event's sources, or a manual confidence demotion, leaves `has_strong_signal` and `confidence` stale, and `has_strong_signal` feeds the confidence scorer. Drift accumulates silently between manual backfill runs.
Verification: `SELECT count(*) FROM events e WHERE e.has_strong_signal = false AND EXISTS (... a strong-signal source ...);` after a backfill window.
Suggested direction: Add a trigger or scheduled recompute, or document that `confidence_backfill.py` must run on a cadence.

### BUG-008 — `source_reliability` refresh silently skipped without `pg_cron`
Severity: P2   Domain: A   Confidence: probable
Location: `packages/db/migrations/0014_schedule_source_reliability_refresh.sql:21-29`
Evidence:
```sql
IF EXISTS (SELECT 1 FROM pg_extension WHERE extname = 'pg_cron') THEN
    PERFORM cron.schedule('refresh-source-reliability', '*/30 * * * *',
                          'REFRESH MATERIALIZED VIEW source_reliability');
ELSE
    RAISE NOTICE 'pg_cron not installed; skipping source_reliability refresh schedule';
END IF;
```
Impact: If the materialized view's refresh schedule is conditionally skipped (pg_cron absent at migration time, or the cluster changes), `verified_rate_30d` / `events_30d` freeze with no warning and the dashboard shows stale reliability. There is no runtime assertion that the schedule exists.
Verification: `SELECT * FROM cron.job WHERE jobname = 'refresh-source-reliability';` and compare `source_reliability` freshness to base-table activity.
Suggested direction: Add an integrity check (in `checks.py`) that the cron job exists and the MV is fresh, or run the refresh from the Python scheduler as a guaranteed fallback.

### BUG-017 — Source-health stamp failure is swallowed
Severity: P3   Domain: E   Confidence: speculative
Location: `apps/ingest/sentinel/jobs/ingest_source.py:109-116`
Evidence:
```python
def _stamp(conn, source, *, posts_inserted, meta):
    try:
        record_source_fetch(conn, source["id"], posts_inserted=posts_inserted, meta=meta)
        conn.commit()
    except Exception as exc:
        conn.rollback()
        log.warning("source_health_stamp_failed", source=source.get("handle"), error=str(exc))
```
Impact: A serialization/format error while stamping health rolls back and only logs a warning; the source's health columns silently stay stale. The isolation from the raw-posts transaction is intentional and correct, but the failure is otherwise invisible outside logs.
Verification: Inject a non-serializable value into `last_fetch_meta` in a test and confirm health columns are unchanged with only a warning.
Suggested direction: Surface repeated stamp failures (counter/alert) so a persistently stale health row is detectable.

### BUG-018 — Code depends on migration-introduced columns with no pre-deploy gate
Severity: P3   Domain: C   Confidence: confirmed (verified currently in sync)
Location: `apps/ingest/sentinel/db.py:355,371` (insert_event), `:548,561` (find_nearby_events), `:604,612` (record_dedup_decision); columns from migrations 0025/0026
Evidence: `insert_event` INSERTs `geocode_precision`; `find_nearby_events` SELECTs and filters on `geocode_precision`; `record_dedup_decision` INSERTs `incoming_precision`/`matched_precision`. These columns come from migrations 0025/0026.
Impact: If the worker code is ever deployed against a DB where these migrations have **not** been applied, every extraction insert and dedup lookup throws `UndefinedColumn` and the pipeline halts. **This is not currently the case** — read-only verification confirms `schema_migrations` records 0024–0027 and all three columns exist in production. The residual finding is the absence of a deploy-time guard that migrations are applied before code that requires them (made more dangerous by BUG-003's bootstrap behavior). Logged at P3 because it is presently in sync; this corrects the ticket's stated (stale) assumption that 0025–0027 were unapplied.
Verification: `SELECT column_name FROM information_schema.columns WHERE table_name='events' AND column_name='geocode_precision';` → returns the column (confirmed present).
Suggested direction: Add a preflight/`checks.py` assertion that required columns exist before the worker processes events; gate deploys on `migrate.py` success.
