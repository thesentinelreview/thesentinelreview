## Area 0 — Data integrity & coverage *(from Claude's read-only Supabase sweep)*

- **DATA-1 — P1 · data/dedup · HEADLINE.** Event duplication / dedup
  fragmentation. Evidence (live 2026-05-30): of 764 events, **133 sit in 44
  same-type/same-day/same-centroid clusters; 89 are collapsible excess
  (~12% of the store is duplication)**. Worst clusters all `strike` / `Kyiv`
  on the identical centroid `POINT(30.5234 50.4501)`: **16 events on
  05-24** (00:00→22:00), **8 on 05-23**, **7 on 05-25**; plus Dnipro 4 on
  05-20. These are facets of single overnight attacks (mostly
  `afp_ukraine_rss` hour-by-hour); only ~2 of the 05-24 set have precise
  coords and are genuinely distinct. Root cause — **NOT just the 6h
  window.** Yes, `find_nearby_events` candidate window `within_hours=6.0`
  (`apps/ingest/sentinel/db.py`) ages rows out during >6h attacks. BUT the
  05-23 cluster (8 events) spans only 4h20m — *inside* the window — and
  still didn't merge. So same-centroid/same-type events are not being
  recognized as the same event even when they're within the time window.
  The fix has to address the **dedup match criteria** (the proximity /
  identity test in `find_duplicate`), not just widen `within_hours`.
  Centroid geocoding (location-less reports → city centroid) compounds it.
  Impact: inflated counts (dashboard shows ~16 "Kyiv strikes" for one
  attack); degrades core product; suppresses the confidence ladder (DATA-2
  — these clusters carry 0–2 `verified`). Fix: rework the
  same-centroid/same-type/same-day merge logic (add a merge pass if needed)
  **while preserving #168/#169** — one careful ticket, NOT a knob change.
  Baseline to beat: **collapsible_excess = 89** (drive down without
  over-merging distinct events). Effort: **M**.

- **DATA-2 — P2 · data.** Confidence ladder works but `verified` is rare.
  Evidence: unconfirmed 572 / partial 178 / **verified 12** / 762 total
  (top tier is `verified`, not `confirmed`). Impact: only 1.6% reach
  `verified`, largely a *consequence* of DATA-1 (fragmented reports never
  accumulate corroboration). Fix: none directly — resolves with DATA-1.
  Effort: **none**.

- **DATA-3 — P2 · data/sources.** `last_post_at` / `health_status` is a
  confounded liveness metric. Evidence: 39 active sources have
  `last_post_at IS NULL` but only **25 have truly never captured** (bluesky
  15, rss 6, x 3, gdelt 1, telegram 0); `silent_active_sources` counts 27 —
  three different numbers. Cause: #175 stamps forward only, no backfill, so
  a source with historical posts reads NULL. Impact: misleads triage and the
  integrity check ("X silent" looked alarming, but X has 919 posts
  all-time). Fix: one-time backfill `last_post_at = max(posted_at)` per
  source from raw_posts, OR repoint `silent_active_sources` at raw_posts.
  Effort: **S**. *(Pairs with A5-3.)*

- **DATA-5 — P2 · data/observability+cost · NEW.** `skip_reason` is
  free-text, not categorical. Evidence (live): **5,839 distinct
  `skip_reason` values**; only TWO are real categories (`off_theater` 258,
  `incomplete_extraction` 51). The other **6,003 skips (95% of all skips)
  carry bespoke LLM-generated prose** as the reason — meaning every one of
  those posts was sent to the extraction LLM and then discarded. Two
  consequences: (a) **observability** — `checks.py` cannot bucket skips
  because each is a unique string, so "why are we skipping" is
  unanswerable; (b) **cost** — confirms the LLM-spend story (see broadened
  A4-2). Fix: make `skip_reason` a small categorical enum set by the
  pipeline (`off_theater` / `off_topic` / `incomplete_extraction` /
  `duplicate` / `low_relevance` / …), and reserve free-text for a separate
  detail column. Effort: **S**. Pairs with A4-2.

- **Clean bill of health (state, no action):** 0 orphaned events · 0 null
  `occurred_at` / `geometry` / `location_name` · 0 future-dated · 0 pending
  raw_posts · stale-date cleanup complete (only the intentionally-kept
  Donetsk KIA remains >14d). **764 events · 7,258 raw_posts (946
  event-producing, 6,312 skipped) · 67 active sources** (as of 2026-05-30
  16:49 UTC).

## Area 2 — Bugs & correctness

- **A2-1 — P0 · S.** `apps/web/lib/queries.ts:1194` queries `s.theater = $1`
  but `sources.theater` was dropped in migration 0012 and replaced with
  `theaters TEXT[]`. Silent `catch { return ... }` at queries.ts:1226 swallows
  the runtime error → `getFirehosePosts` always returns empty. Fix:
  `$1 = ANY(s.theaters)` (mirrors `getSensorStripData` queries.ts:578).
  **Verify-first gate:** confirm the firehose is a live, user-facing route
  by hitting it before locking P0; if it's not a shipped surface, downgrade.
- **A2-2 — P1 · M.** Duplicate `0009_*` migration prefix:
  `packages/db/migrations/0009_stripe_webhook_idempotency.sql` and
  `0009_watches.sql`. `migrate.py` alphabetical sort works today
  (`stripe` < `watches`), but the prefix is no longer unique — a future
  `0009_a*.sql` would silently reorder the apply sequence. Rename / renumber
  in a follow-up migration (with a `schema_migrations` rename) or document
  the break in CLAUDE.md.
- **A2-3 — P1 · M.** `packages/db/migrate.py:54-67` bootstrap path marks
  every on-disk migration except the LAST as already-applied when
  `schema_migrations` is empty but `sources` exists. A fresh disaster-recovery
  DB hitting this path silently skips 21 migrations and tries to apply only
  the newest. Fix: mark ALL on-disk files applied during bootstrap, or refuse
  to bootstrap without an explicit `--bootstrap` flag.
- **A2-5 — P1 · S.** `apps/ingest/sentinel/runner.py:48-89` `_drain_queue`
  raises `_AllJobsFailed` when every *processed* job failed; with one job in
  the run, one failure trips a full-pipeline alert. Bar should be
  `len(processed) >= N AND all failed` (e.g. N = 3).
- **A2-6 — P2 · S.** `apps/ingest/sentinel/db.py:130-163` `_classify_fetch`
  returns a positional 4-tuple `(health, detail, newest, is_error)`; easy
  to misuse. NamedTuple / dataclass would prevent future bugs.
- **A2-7 — P2 · S.** `apps/ingest/sentinel/pipeline/translator.py:200-204`
  flags but accepts translations with suspicious length ratio. Long-tail
  risk: the LLM occasionally truncates non-Latin scripts, producing a short
  English "translation" that loses content. Watch-metric now; consider a
  hard ratio floor (return NULL) later.
- **A2-8 — P1 · S.** `apps/ingest/sentinel/jobs/extract_events.py:199-202`
  `incomplete_extraction` silently skips when any required field is None
  (e.g. `has_event=true` but missing oblast or lat/lng) → real loss of
  events. **Measured (live): 51 lifetime, 29 in the last 7d, 6 in 24h — vs
  698 event-producing posts in 7d, so ~4% of would-be events are being
  silently dropped.** Confirmed material. Surface the counter via a
  `checks.py` rate check; consider letting oblast-only-missing through with
  `oblast='Unknown'`.

## Area 4 — Pipeline quality

- **A4-1 — P1 · S.** `apps/ingest/sentinel/checks.py:195`
  `passed = skip_pct <= 75`; production is ~74% (24h) and 87% lifetime —
  the warning sits on the threshold and trips on small fluctuations. Raise
  to 80, add hysteresis (warn 75 / alert 85), or alert on per-theater skip
  rates. Cross-refs DATA / PIPE-1.
- **A4-2 — P1 · M [BROADENED — not GDELT-only].** Nearly all skips burn
  the extraction LLM. Measured (live): **6,003 of 6,312 skips (95%) carry
  post-LLM free-text reasons** (see DATA-5) — i.e. those posts were sent
  to the LLM and discarded. Attribution: **GDELT 2,431 (~40%** —
  corroborates the prior "GDELT ~41% of spend"); **non-GDELT off-topic
  3,572 (~60%)**. So a GDELT-only CAMEO allowlist fixes only ~40% of the
  waste. Fix: add a **cheap pre-LLM relevance gate** (keyword / theater /
  CAMEO heuristic, or a small-model classifier) that drops obvious
  off-topic posts *before* the expensive extraction call — covering both
  GDELT and the non-GDELT long tail. Effort: M. Implements alongside
  DATA-5's categorical `skip_reason`.
- **A4-3 — P2 · S.** `apps/ingest/sentinel/pipeline/theater_router.py:75`
  builds a single-theater prior even for multi-theater sources

- **A4-4 — P2 · S.** Translations backfill workflow
  (`.github/workflows/sentinel-backfill-translations.yml`) lacks a `dryrun`

- **A4-5 — P1 · S.** `relevance_score` is marked `required` in the tool
