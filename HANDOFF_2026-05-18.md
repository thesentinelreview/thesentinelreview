## Part 1 — Dashboard Data Audit (8 bugs fixed)

### C1: Briefing generator ignored the `theater` parameter

**Files:** `apps/ingest/sentinel/db.py`, `apps/ingest/sentinel/jobs/generate_briefing.py`  
**Commit:** `c5d1b86`

`get_recent_events()` accepted a `theater` argument but never used it in the SQL `WHERE` clause.
Ukraine briefings contained Sudan and Myanmar events. `_compute_baseline()` had the same problem —
the 7-day event baseline mixed all theaters.

Fixed by adding a PostGIS bounding-box filter to both functions using the same `_THEATER_BBOX`
coordinates as the frontend:

```python
_THEATER_BBOX = {
    "ukraine": (22, 44, 40, 52),
    "iran":    (44, 25, 64, 40),
    "sudan":   (21,  8, 42, 23),
    "myanmar": (92,  9, 102, 29),
}
```

### C2: `is_high_impact` silently defaulted to `False`

**File:** `apps/ingest/sentinel/pipeline/extractor.py`  
**Commit:** `c5d1b86`

The LLM tool schema did not list `is_high_impact` in its `"required"` array. If the model
omitted the field, `raw.get("is_high_impact", False)` silently defaulted to `False`, bypassing
the (then-active) human review gate for events that should have been held.

Fixed by adding `"is_high_impact"` to the required fields in `_TOOL`. The field is retained in
the schema for analytics even though the hold gate has since been removed (see Part 3).

---

### C3: `getAllSources()` showed 24-hour data in the "7-day" column

**File:** `apps/web/lib/queries.ts` → `getAllSources()`  
**Commit:** `c5d1b86`

Copy-paste bug on line 706: `events_7d: Number(r.events_today) || 0`. The 7-day column on the
Sources page was always identical to the Today column.

Fixed by adding a separate `week` subquery:

```sql
LEFT JOIN (
  SELECT es.source_id, COUNT(DISTINCT es.event_id) AS cnt
  FROM event_sources es
  JOIN events e ON e.id = es.event_id
  WHERE e.occurred_at > now() - INTERVAL '7 days'
    AND e.published_at IS NOT NULL
  GROUP BY es.source_id
) week ON week.source_id = s.id
```

and assigning `events_7d: Number(r.events_7d) || 0`.

### C4: `getStats()` fell back to placeholder data when a time window had 0 events

**File:** `apps/web/lib/queries.ts` → `getStats()`  
**Commit:** `c5d1b86`

```typescript
if (!row || Number(row.events) === 0) return ph.phStats(theater);
```

When a theater had no events in the selected 24h/7d/30d window, the stats panel showed
fabricated placeholder numbers — indistinguishable from real data to the user.

Fixed to return real zeros instead:

```typescript
if (!row) return ph.phStats(theater);
if (Number(row.events) === 0) return { events: 0, strikes: 0, verified_pct: 0, vs_7d_avg_pct: 0 };
```

### H2: Myanmar `is_high_impact` threshold was far too broad

**File:** `apps/ingest/sentinel/pipeline/extractor.py`  
**Commit:** `c5d1b86`

Myanmar's system prompt flagged *"airstrikes on civilian infrastructure (schools, hospitals,
markets)"* and *"capture of a significant town"* as high-impact — events that occur multiple
times per day in the Myanmar conflict. Ukraine's equivalent threshold requires 10+ mass
casualties, WMD, or a cross-border escalation involving a new country.

The Myanmar definition was narrowed to match Ukraine:

> `is_high_impact` must be true ONLY for: mass-casualty events (10+ killed/wounded claimed),
> confirmed use of chemical/biological/radiological weapons, or an unprecedented cross-border
> escalation targeting a country not previously struck in this conflict. Routine airstrikes,
> clashes, and territorial changes are NOT high-impact regardless of location.

Note: `held_for_review` has since been eliminated entirely (see Part 3), so this threshold
now only affects analytics tagging.

### H3: `getAlerts()` always showed 24-hour alerts regardless of the selected time range

**File:** `apps/web/lib/queries.ts` → `getAlerts()`  
**Commit:** `c5d1b86`

The function hardcoded `INTERVAL '24 hours'` in its WHERE clause and accepted no `timeRange`
parameter. When a user switched to 7d or 30d view, the alert strip stayed on 24h.

Fixed by adding `timeRange: TimeRange = "24h"` as a third parameter and interpolating
`SQL_INTERVALS[timeRange]`. Both `apps/web/app/page.tsx` and `apps/web/app/app/page.tsx` were
updated to pass `timeRange` to the call.

### H4: `getTopSources()` showed placeholder data on quiet days

**File:** `apps/web/lib/queries.ts` → `getTopSources()`  
**Commit:** `c5d1b86`

`WHERE s.is_active = true AND COALESCE(today.cnt, 0) > 0` — any day with no events in the
last 24 hours returned an empty result set, which triggered the placeholder source list.

Fixed by removing the `> 0` condition. Active sources are always shown, ranked by 30-day
historical activity. Sources with `events_today = 0` are real and still useful to display.

## Part 2 — Data Integrity Failsafe

**Commit:** `375ae60`

Ingest runs on a 30-minute GitHub Actions schedule. GitHub only emails on *workflow failure*,
not on silent data degradation (e.g. all posts skipped, LLM extraction broken, DB writes
silently failing). This adds three layers of continuous monitoring.

### New: `sentinel-run-checks` entry point

**File:** `apps/ingest/sentinel/checks.py`

Ten integrity checks across two severity levels, each querying the database directly:

**Critical** (exit 1 → GitHub Actions marks job failed → GH sends failure email to repo owner):

| Check | What it detects |
|---|---|
| `stuck_running_jobs` | Jobs stuck in `status='running'` for >30 min (worker crashed mid-job) |
| `failed_jobs_24h` | Jobs that exhausted all retry attempts in the last 24h |
| `no_published_events_48h` | No event published in last 48h — pipeline stopped producing output |
| `orphaned_published` | Events with `published_at IS NULL` and `held_for_review = false` (migration regression) |
| `future_occurred_at` | Events with `occurred_at > now() + 1h` (LLM extracted a future date) |

**Warning** (logged and reported to webhook/email, but no exit 1):

| Check | What it detects |
|---|---|
| `unprocessed_posts_old` | Raw posts ingested >2h ago still unprocessed (extraction queue stuck) |
| `high_skip_rate_24h` | >75% of posts skipped in last 24h (LLM prompt or source quality issue) |
| `no_published_events_8h` | No event in 8h — early warning before the 48h critical threshold |
| `held_events` | Any event with `held_for_review = true` (policy violation since Part 3) |
| `silent_active_sources` | >5 active sources with no posts ingested in the last 72h |

**File:** `apps/ingest/sentinel/runner.py` — `run_checks()` function added.

**Entry point:** `sentinel-run-checks` added to `apps/ingest/pyproject.toml`.

### Alerting

The check runner sends alerts via two optional channels when critical failures are found:

**Email (SMTP):** Set all four env vars to activate:

```
ALERT_EMAIL=you@example.com
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=alerts@yourdomain.com
SMTP_PASSWORD=...
SMTP_FROM=                     # optional override of From address
```

Subject: `[Sentinel] Integrity check FAILED — N critical failure(s)`  
Body: plain-text list of failed check names and details, with a link to the Actions log and
`GET /api/admin/integrity`.

**Slack webhook:** Set `SENTINEL_ALERT_WEBHOOK_URL` to a Slack incoming webhook URL. Sends a
formatted message with the same content.

Both channels are triggered by the same critical-failure condition and fire together if both are
configured. Neither fires on warnings-only runs.

### GitHub Actions integration

**File:** `.github/workflows/sentinel-ingest.yml` — new `Run integrity checks` step added
after each 30-min ingest run. If checks fail, the ingest job itself is marked failed.

**File:** `.github/workflows/sentinel-integrity.yml` — new standalone workflow running every
2 hours at :15 past, independent of ingest. Provides coverage during ingest scheduling gaps.

### Web endpoint

**File:** `apps/web/app/api/admin/integrity/route.ts`

`GET /api/admin/integrity` runs all ten checks and returns structured JSON:

```json
{
  "ok": false,
  "critical_failures": 1,
  "warnings": 0,
  "checked_at": "2026-05-18T01:00:00.000Z",
  "checks": [
    {"name": "stuck_running_jobs", "passed": true, "severity": "critical", "detail": "0 stuck jobs", "value": 0},
    {"name": "failed_jobs_24h", "passed": false, "severity": "critical", "detail": "2 job(s) permanently failed in last 24h (types: extract_events)", "value": 2}
  ]
}
```

## Part 3 — Removed Human Review Gate

**Commit:** `20700c7`

The dashboard is fully autonomous. Nothing should ever be withheld from publication for human
review.

### scorer.py

**File:** `apps/ingest/sentinel/pipeline/scorer.py`

`held_for_review` is now always `False` regardless of `is_high_impact`:

```python
# Before
held = is_high_impact

# After
held = False  # Dashboard runs autonomously — no human review gate.
```

The `is_high_impact` field is retained in the LLM tool schema for analytics/future use, but
it no longer affects the publishing pipeline.

### Migration 0007 — publish existing held events

**File:** `packages/db/migrations/0007_publish_held_events.sql`

Seven events were stuck with `held_for_review = true` and `published_at = NULL`. This migration
publishes them:

```sql
UPDATE events
SET published_at    = COALESCE(published_at, created_at),
    held_for_review = false
WHERE held_for_review = true;
```

Applied automatically on the next ingest run (the ingest workflow calls `migrate.py` before
each ingest cycle).

### Integrity check updated

The `held_backlog_old` check (which waited 48h before flagging held events) was renamed to
`held_events` and now flags **any** held event immediately, since `held_for_review = true` is
a policy violation going forward.

## Remaining Known Issues

### `occurred_at` approximated for pre-fix events

Events corrected by migration 0006 (previous session) have `occurred_at = created_at` rather
than the true conflict event time. New events ingested after the extractor fix have accurate
timestamps. The historical approximations will have decreasing impact as the dataset grows.

### No briefing for non-Ukraine theaters yet

Briefings are generated twice daily (12:00 and 20:00 UTC) for all theaters. However, Sudan and
Myanmar sources were added relatively recently and may not have enough verified/partial events
in the 24h window for the briefing generator to produce output. This resolves naturally as more
events accumulate.

### Slack webhook and SMTP not yet configured

Both channels are implemented and ready. Secrets need to be added to the GitHub repository
(Settings → Secrets and variables → Actions):

- `SENTINEL_ALERT_WEBHOOK_URL` — Slack incoming webhook
- `ALERT_EMAIL`, `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD` — email alerting
